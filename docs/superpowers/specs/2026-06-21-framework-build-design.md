# 設計：新增 skill `framework-build`（理論框架定錨）

> 日期：2026-06-21
> 範圍：**新增一個獨立 skill**，插在 `lit-matrix`（讀文獻）與 `method-design`（選方法）之間，並重編 `research-roadmap` 階段、把 `outline-builder` 那關更名。涵蓋三個模式：正向定錨 ＋ 輔助框架嵌入 ＋ 逆向體檢。

## 背景

在另一台電腦跑真實場景（一篇關於日台半導體經濟安全／JASM 戰略耦合的社科課題）時，暴露一個重大缺口：**整套 Boya 流程從 `lit-matrix`（讀文獻）→ `method-design`（選方法）→ `outline-builder`（搭骨架），中間沒有任何一步幫研究者把「我用什麼理論透鏡看這整個問題、關鍵概念是什麼、概念之間怎麼連」講清楚。**

真實場景中，這一步其實被做了——agent 在「關 3.5」即興攤開了四個候選框架（地緣經濟學、武器化相互依賴、安全化、戰略耦合）、標了每個的理論代價與「庫存支撐」、推薦了分層結構、停在 GATE 讓作者拍板主框架，並在作者追問「需不需要准同盟理論」時把它定位成桥接性辅助框架——產出了一個 `03b_理论框架定锚.md`。

問題是：**這一切是 agent 臨場發揮，沒有任何 skill 在背後撐**——沒有 `SKILL.md`、沒有 `evals/` 守、沒進 `ROUTER.md`、`research-roadmap` 全流程總表也沒這一關。換一台電腦、換一個 session 就可能不穩定或不發生。

## 問題

1. **缺一個 home。** 理論框架定錨沒有對應 skill。研究者問「我這篇要用什麼理論框架」時，ROUTER 無法路由，roadmap 無此階段。
2. **roadmap 階段名不符實。** `research-roadmap` 全流程總表第 5 階段叫「**框架與大綱**」，但對應 skill 只有 `outline-builder`，而它從頭到尾只處理大綱／章節骨架／論證鏈，完全不碰理論框架。階段名承諾了「框架」，產出物只有「骨架」。
3. **即興發揮不可複現。** 真實場景做得很好的那套工法（不堆框架沙拉、庫存支撐誠實標註、硬 GATE、差異化 vs 紅海、桥接辅助層）沒有被固化成規則與 eval，無法保證下次重現。

## 目標

新增 skill `framework-build`（中文：理論框架定錨），插在 `lit-matrix`↓ 與 `method-design`↑ 之間（因為**框架從文獻長出、又往下被操作化成方法**），把真實場景驗證過的工法固化成鐵律與 eval。涵蓋三模式：

1. **正向定錨**（主流程）：從文獻地圖萃取候選框架 → 攤代價 → 推薦分層 → 硬 GATE。
2. **輔助框架嵌入**（子模式）：處理「我還要不要用理論 X」——判 X 該當主框架還桥接層、嵌哪一層、會不會喧賓奪主。
3. **逆向體檢**（子模式，對齊 `outline-builder` 的逆向體檢）：拿已寫好的框架反推它撐不撐得住、有沒有兩張皮、對不對得上研究問題。

## 非目標（明確不做）

- **不替研究者決定主框架。** AI 攤候選＋給推薦，但採不採納是硬 GATE，永遠是研究者的。
- **不編造框架的承重文獻。** 庫裡沒有的框架一律標「需補 X 篇」交回去找真文獻，絕不假裝有地基（呼應 `citation-verify`）。
- **不接管 `method-design` 的活。** 框架定錨完只指出每個概念的「實證抓手候選」，真正操作化成方法／指標／訪談題是 `method-design`。
- **不接管 `outline-builder` 的活。** 框架不負責長章節骨架；它產出的是「透鏡與分層」，骨架交給 `outline-builder`。
- **不代筆框架段落。** 逆向體檢只診斷，改寫交 `style-tune`。

## 設計

### ① skill 命名與定位

- **name**：`framework-build`（kebab-case，動詞結尾，與 `outline-builder`／`method-design` 同構）。
- **中文顯示名**：`理論框架定錨`。
- **階段**：`research-roadmap` 全流程的新階段，落在 `lit-matrix` 與 `method-design` 之間。

### ② 三模式與觸發語（寫進 description）

| 模式 | 何時用 | 觸發語 |
|---|---|---|
| 正向定錨 | 文獻讀完、要定理論框架（寫之前） | 「我這篇要用什麼理論框架」「幫我定錨理論框架」「有哪些框架可以套」「主框架該選哪個」 |
| 輔助框架嵌入 | 已有主框架、糾結要不要加某理論 | 「我還要不要用理論 X」「要不要加上 Y 理論」「這個理論放主框架還是輔助」 |
| 逆向體檢 | 框架已寫好、要檢查撐不撐得住 | 「幫我看我寫好的這段框架撐不撐得住」「框架跟研究問題對不對得上」「有沒有兩張皮」「框架是不是太雜」 |

### ③ 鐵律（eval 守這幾條）

1. **框架從文獻長出，不從記憶編。** 候選框架必須能對回研究者手上已有的文獻（`lit-matrix` 產出）。AI 可基於通識「提名」一個庫裡沒有的框架，但必須誠實標「庫存支撐」（✅有／🟡部分／❌需補 X 篇），對沒地基的一律標「需補」並交回研究者找真文獻，絕不假裝已有地基。（呼應 `citation-verify`：框架的承重文獻也要是真的。）
2. **攤候選＋給推薦，但主框架選擇權是你的（硬 GATE）。** 這跟 `topic-refine`「不替你選題」不衝突——框架的代價是可攤開、可陳述的事實（理論代價、庫存支撐、紅海風險），AI 在可分析的地基上給推薦；但「採不採納」藏著學科品味與卡位戰略，是研究者的判斷。GATE 必停，給「採納推薦／保守最小／你另有偏好」三條路，拍板才接力 `method-design`。
3. **不堆框架沙拉。** 主框架最多一個；輔助／中介框架要有明確分工與「為什麼需要它」。每多一個框架，都要回答「它幹的活前面的框架幹不了嗎」，答不出就砍。
4. **框架要對得上研究問題與差異化卡位。** 框架是「我用什麼透鏡看這整個問題」，不是裝飾。選框架時盯：它解釋的東西跟研究問題對不對得上？選它會不會把研究推回擁擠紅海、丟掉差異化？（真實案例：選准同盟當主框架會滑回「日台安全」紅海。）
5. **框架要能往下操作化。** 定錨完的框架要能交給 `method-design` 變指標、交給 `outline-builder` 長章節。一個無法操作化、無法落到實證抓手的框架是空的。報告結尾要為每個關鍵概念指出「實證抓手候選」。
6. **分學科分流，不一套打天下。** 實證／社科型要的是「概念框架」（關鍵概念—變項—命題—關係模型）；人文思辨型要的是「理論視角／透鏡」（選一個學派／理論家貫穿詮釋）。先判論文類型再給對應的定錨方式，像 `outline-builder` 不預設 IMRaD 一樣。

### ④ 正向定錨工作流

1. **盤點手上材料。** 確認研究者已有：研究問題（含類型：描述／解釋／評價）、`lit-matrix` 文獻地圖（共識／分歧／缺口）、論文類型。缺就回前面 skill。
2. **判論文類型 → 選定錨路徑。** 實證型走「概念框架」、思辨型走「理論視角」（鐵律 6）。
3. **從文獻地圖萃取候選框架。** 含 AI 基於通識提名的庫外框架，逐個誠實標庫存支撐（鐵律 1）。
4. **攤候選代價表**（四欄，見輸出）。
5. **推薦分層結構＋理由**（主框架 → 中介機制 → 實證抓手 → 落點）。
6. **GATE：交回研究者拍板主框架 ＋ 待補清單。**
7. **拍板後**：輸出定錨表，接力 `method-design`（把框架操作化）。

### ⑤ 輔助框架嵌入子模式

觸發：「我還要不要用理論 X」。做法：
- 判 X 該當主框架還是桥接／辅助層。
- 給「主 vs 辅助」定位 ＋ 代價（會不會滑回紅海、會不會喧賓奪主）。
- 若採辅助，明確它承擔哪 1–2 個具體任務、嵌在分層結構哪一層。
- 標庫存支撐（鐵律 1），GATE 同樣交回（鐵律 2）。

### ⑥ 逆向體檢子模式

觸發：「幫我看我寫好的這段框架撐不撐得住」。做法：從已寫好的框架文字反推它的分層與承重文獻，逐項體檢：

| 體檢項 | 問法 |
|---|---|
| 承重文獻真偽 | 框架引的承重文獻真實存在嗎？（提醒走 `citation-verify`） |
| 對得上研究問題 | 這個透鏡解釋的東西，扣得回研究問題嗎？ |
| 有沒有兩張皮 | 框架說一套、實證／方法做一套？ |
| 有沒有框架沙拉 | 層數是不是貪多？每層分工說得清嗎？ |
| 實證抓手 | 每個關鍵概念有對得上的實證抓手嗎？ |

只診斷不代筆；改寫交 `style-tune`、整篇審稿交 `self-review`。

### ⑦ 產出物：理論框架定錨表（存 `框架定錨.md`）

```markdown
# 理論框架定錨：<研究問題簡述>

## 一、候選框架攤牌
| 框架 | 解釋什麼 | 理論代價 | 庫存支撐 |
|---|---|---|---|
| 地緣經濟學 | 經濟工具達成地緣目的 | 概念偏寬易成筐 | ✅ Katada／CSIS |
| 戰略耦合 | 市場分工→國家戰略耦合 | 庫裡沒有 | 🟡 需補 2 篇 Yeung&Coe |

## 二、推薦分層（理由）
主框架  <…>            ← 為什麼是它
  ↓
中介機制 <…>            ← 差異化核心
  ↓
實證抓手 <…>            ← 可量化／可操作
  ↓
落點    <…>

## 三、🚦 GATE — 只有你能定主框架
- 採納推薦：<…>
- 保守最小：<…>
- 你另有偏好：<…>

## 四、待補清單
| 框架 | 缺什麼 | 回哪個 skill |
|---|---|---|
| 戰略耦合 | 承重文獻 2 篇 | citation-verify / lit-matrix |
```

可選搭配 work-trace-table（CONVENTIONS §9）：候選框架表的「庫存支撐」用 ✅ 已查證／❓ 待查 標記，主框架選擇用 🧍 你判斷。

### ⑧ 邊界（與 Boya 立場一致）

- **主框架是研究者的**（鐵律 2，硬 GATE），對齊 `topic-refine`／roadmap「不替你做研究判斷」。
- **不編承重文獻**（鐵律 1），對齊 `citation-verify`／RULES.md「查無≠偽造」。
- **DRY 分工**：操作化指回 `method-design`；長章節指回 `outline-builder`；改寫指回 `style-tune`；整篇審稿指回 `self-review`。各以一句交叉指引串接，不重抄。

## 驗收（依 CONVENTIONS §4）

- `framework-build` 升版號前，須有 ≥1 篇 `examples/` 實跑真錄（用脫敏後的 JASM 對話），展示定錨表被正確產出、GATE 有停、庫存支撐有誠實標註。
- 回歸斷言寫進 `evals/framework-build.md`，至少含：
  - **不編承重文獻**：庫外框架必標「需補」，不得假裝已有地基。
  - **硬 GATE**：必停在主框架選擇、給三路、不替研究者拍板。
  - **不堆框架沙拉**：主框架最多一個；每多一層要說得清分工。
  - **可操作化**：每個關鍵概念要指出實證抓手候選。
  - **分學科分流**：實證走概念框架、思辨走理論視角，不一律套同一種。
  - **三模式齊全**：正向定錨／輔助嵌入／逆向體檢都有對應行為。
- 每次實測後 `VERIFICATION.md` 補一行。

## 版號影響

- 四份 README 的 skill 計數 +1。
- `outline-builder` 僅更名其 roadmap 階段標籤、加交叉指引，視情況升 patch；`research-roadmap` 因重編號升 minor。

## 一致性檢查（與既有規約）

- **聚焦不膨脹**：新增 1 個 skill 是真實場景證實的缺口，非速度性增生；它接管的是現況無人負責的一段。
- **DRY**：定錨工法只在 framework-build 定義一次；操作化／骨架／改寫／審稿全用交叉指引指回對應 skill，不重抄。
- **誠信紅線**：鐵律 1（不編承重文獻）對齊 RULES.md「查無≠偽造」；鐵律 2（硬 GATE）對齊「不替你做研究判斷」。
- **分層參考料（§8）**：SKILL.md 先單檔；若三模式＋分學科撐爆 200 行軟警戒線，再把學科細節（概念框架 vs 理論視角的長對照）拆進 `skills/framework-build/references/`。
- **選擇性借用**：工法源自自家真實場景，非外部專案搬運。
